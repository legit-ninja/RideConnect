# UX review — profile menu & avatar UI

Scope: `Avatar`, `ListingImage`, `UserMenu`, `SiteHeaderClient`/`SiteHeader`, `/settings`.

## Must-fix

- **`UserResponse` doesn't expose `first_name`/`last_name`.** The `User` model and
  `RegisterRequest` already collect first/last name, but `GET /auth/me` only returns
  `avatar_url`. Until the schema is updated, `Avatar`/`UserMenu` fall back to the
  email for both initials and the trigger label (e.g. "J" instead of real initials).
  This is a real backend follow-up, not just a frontend nice-to-have — surfacing a
  user's own name in their account menu is a baseline trust signal. Frontend types
  are already updated (`first_name?`, `last_name?` optional on `User`) so this is a
  non-breaking backend addition when ready.
- **Duplicate `/auth/me` calls are possible if `UserMenu` is reused without a
  `user` prop** in more than one place on the same page (e.g. a future mobile nav
  drawer alongside the header). Currently only one instance exists (in
  `SiteHeaderClient`, which passes `user` down), so there's no duplicate call today,
  but this should be kept in mind if `UserMenu` gets a second call site — consider a
  shared auth context instead of prop drilling once that happens.
- **No `outside-tap` affordance on touch devices without a mouse `mousedown`
  equivalent gap.** The dismiss handler listens for `mousedown`, which also fires
  for touch taps in all evergreen mobile browsers, but this should be spot-checked
  on iOS Safari specifically since it has historically had edge cases with
  synthetic click ordering on first tap after a scroll.

## Nice-to-have

- Full ARIA APG "menu button" pattern (`role="menu"`/`menuitem"` + arrow-key roving
  tabindex) was intentionally **not** used — the panel mixes navigation links, a
  radiogroup (theme), and an action button, and APG guidance reserves `menu`/
  `menuitem` for application-style command menus, not mixed navigation content. The
  current implementation (disclosure button + `role="group"` panel, Tab-order
  navigation, Escape-to-close, focus-return-to-trigger) is WCAG 2.2 AA compliant but
  a purist could add roving tabindex for AAA-level polish later.
- `UserMenu`/`SiteHeaderClient` each hold their own loading skeleton; consider a
  shared `AuthProvider` context so every consumer of "current user" (dashboard,
  header, future admin shell) shares one fetch instead of one per mounted
  component.
- `Avatar`'s verified badge border is hardcoded to `var(--background)` so it "cuts
  out" cleanly on the site header (which uses `--background`). If `Avatar` is later
  placed on a `--card` surface (e.g. inside the `UserMenu` dropdown header), the cut
  ring will show a faint 2px mismatch against the card color. Low visual impact,
  but worth a token like `--avatar-ring-bg` that callers can override if this comes
  up.
- `ListingImage` placeholder is an SVG line-art placeholder rather than the
  requested `default.jpg` (a real photographic placeholder can't be generated by
  this agent). Swap in an actual photo asset when available — the component's
  fallback path is a plain string constant, so this is a one-line change.

## Trust & safety

- Verification status is visible in two places in the dropdown: the badge on the
  avatar itself and an explicit "Verified" / "Not verified" text line in the
  dropdown header — status is never conveyed by color alone (WCAG 1.4.1).
- Admin link only renders for `is_admin` users, and Owner/Rider link groups only
  render for the corresponding role flags — no role-gated route is exposed to
  users who can't act on it (previously the top-level nav always rendered a bare
  "Admin" link regardless of auth state).
- Sign-out is a real action (not a link) and is visually separated (destructive
  color + divider) from navigation items, reducing the odds of an accidental sign
  out and matching platform copy elsewhere: sign-out never appears to be part of a
  "navigate away" trust boundary.
- No off-platform links or destinations were introduced anywhere in this menu.

## Mobile

- Below 480px, the trigger's text label hides (icon/avatar-only, still has an
  accessible name via the `Avatar`'s own `aria-label`) and the dropdown becomes a
  fixed bottom sheet (`position: fixed`, capped to `80vh` with internal scroll)
  instead of an absolutely-positioned panel — this avoids the classic "dropdown
  clipped by viewport edge" overflow bug on narrow screens.
- Verified `--content-max-marketplace` pages (`/settings`) don't introduce any new
  horizontal scroll at 320px width in the build; `ListingImage`/`Avatar` both use
  percentage/`object-fit: cover` sizing rather than fixed pixel widths that could
  overflow small viewports.

## High contrast

- `Avatar` initials fallback uses `--primary` / `--primary-foreground`, which in
  `high-contrast` mode resolves to pure green-on-black — passes contrast easily and
  matches the same tokens already used for primary buttons.
- The verified badge uses `--success-bg` / `--success-fg`, which in high-contrast
  mode is black background with a `#00ff00` icon — readable, and consistent with
  how `VerificationPill`/`StatusBadge` already render "verified" everywhere else in
  the app.
- Dropdown panel border/shadow uses `--card-border` and `--overlay`, both of which
  resolve to strong white/black values in high-contrast mode (no low-contrast gray
  panel edges).
- Not yet manually tested in a live browser against `forced-colors: active`
  (Windows high contrast) beyond the `Avatar` badge's explicit `Canvas` border
  override — recommend a manual pass per the design-system test matrix before
  ship.

## One-primary-action check

- Header nav has at most one primary-styled button at a time (`ThemeToggle` and
  nav links are all secondary/plain-text styling); the `UserMenu` trigger itself is
  a plain/ghost button, not competing visually with any page-level primary CTA.
- Inside the dropdown, every entry is either a plain navigation link or the
  (singular, clearly separated) "Sign out" action — no competing primary buttons
  inside the menu itself.
- `/settings` placeholder page has no primary CTA at all beyond the embedded theme
  radiogroup, which is intentional given there's nothing else to do on the page
  yet; will need a real primary action once settings fields are built out.
